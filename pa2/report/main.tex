\begin{abstract}
In this assignment, we created a pipeline for extracting and indexing textual content from
previously crawled web pages to support domain-specific information retrieval.
The pipeline parses raw HTML and retrieves semantically coherent chunks of data (article headings, body, etc.), 
which are then encoded into dense vector representations using a pre-trained language model.
The resulting embeddings are persisted in a PostgreSQL database extended with the
pgvector module, enabling efficient similarity search over the document collection.
\end{abstract}

\section{Introduction}
This report describes the design and implementation of a parsing and semantic embedding
pipeline that processes the HTML pages collected by the preferential web crawler developed
in the previous assignment. The goal is to extract clean article text from the crawled
pages and represent it in a form that supports efficient domain-specific information retrieval.

The pipeline consists of two main stages: preprocessing and retrieval. During preprocessing, crawled HTML pages from 24ur.com are parsed, and the extracted text is divided into sentence-based chunks. The resulting chunks are then embedded using a transformer-based sentence embedding model and stored in a vector database.

During retrieval, a user query is embedded using the same embedding model and used to retrieve the most similar chunks from the database. Query--chunk pairs are subsequently evaluated by a reranking model, which refines the initial ranking based on semantic relevance. The final output of the pipeline is a ranked list of text chunks that are most semantically relevant to the query.





\section{Methodology}



\subsection{Web Page Parsing}

Parsing is implemented with the \texttt{lxml} library and targets the article body of each page by
locating known structural elements such as the \texttt{\#article-body} block. Article titles
are recovered from OpenGraph metadata where available. The chunker uses NLTK's Slovene
sentence tokenizer to split text into sentence-boundary-aware chunks with a configurable
maximum word count.

\subsection{Chunking}

After parsing the crawled web pages, the extracted text is divided into smaller sentence-based chunks. Sentence boundaries are detected using the NLTK sentence tokenizer configured for the Slovene language. Chunks are constructed incrementally by appending consecutive sentences until a predefined maximum word limit is reached. Once the limit is exceeded, a new chunk is created. This approach preserves local semantic context while ensuring that chunk sizes remain suitable for transformer-based embedding models.







\subsection{Chunk Embedding}

Document chunks were transformed into dense vector representations using transformer-based sentence embedding models. Each chunk was embedded independently, and the resulting vectors were stored in dedicated database tables grouped by embedding dimensionality.

To ensure consistent similarity computation during retrieval, embeddings generated by all evaluated models were normalized to unit length prior to storage. An exception was made for the \texttt{BAAI/bge-m3} model, for which both normalized and unnormalized embeddings were computed in order to analyze the influence of normalization on retrieval metrics and retrieval quality.

The embedding computation included both multilingual and Slovenian-specialized embedding models. Table~\ref{tab:embedding_models} summarizes the embedding models used in the experiments together with their output vector dimensionalities.

To support efficient similarity-based retrieval, vector indexes were created for each embedding table using the \texttt{pgvector} PostgreSQL extension. Separate HNSW indexes were generated for cosine similarity and Euclidean (L2) distance metrics, enabling experimentation with different retrieval strategies. Additionally, standard indexes were created for embedding model identifiers to enable efficient filtering by model.







\subsection{Chunk Retrieval and Reranking}

During retrieval, the user query is embedded using the same embedding model as the stored chunk embeddings. The generated query vector is then used to query the vector database and retrieve the top 100 most similar chunks according to the selected distance metric. The evaluated retrieval metrics included cosine similarity, Euclidean (L2), and Manhattan (L1) distance. Since \texttt{pgvector} does not provide HNSW indexing support for L1 distance, L1 retrieval is performed without vector indexing.

The retrieved chunks are subsequently reranked using a cross-encoder model. For reranking, query--chunk text pairs are constructed and processed jointly by the reranker model to produce relevance scores. The final retrieval output consists of the 10 highest-ranked chunks according to the reranker scores.

\section{Embedding Analysis}

To evaluate the retrieval quality of our setups, we employ the Normalized Discounted Cumulative Gain metric at \(k\) retrieved chunks (NDCG@\(k\)). This metric is well suited for retrieval pipelines, as it measures not only the number of relevant chunks among the top \(k\) retrieved results, but also whether the most relevant chunks appear near the top of the ranking. For evaluation, we compute the mean NDCG@\(k\) across multiple queries, providing a more representative measure of average retrieval performance.

Queries used when computing mean NDCG@k scores, were obtained by randomly picking some number of pages from the database and taking the first segment of each page, where only the first 128 characters were used. After we retrieved N closest segments to each query (for each model), ground truth relevancy score was calculated by summing scores of two strategies:

\begin{enumerate}
    \item Okapi BM25 contributes to the first 50\% of the score. This ranking function was chosen due to its popularity among search engines.
    \item The other 50\% is 1 if a segment appears on the same page as the query, 0 otherwise. 
\end{enumerate}

The 50\% weights for both strategies were chosen arbitrarily.

First, we evaluated the quality of the embedding models by retrieving N closest segment embeddings to each of the M query embeddings according to the used distance metric. This provides a baseline performance measure for selecting the best-performing embedding model without introducing bias from reranking models.

The embedding model performance is shown in Figure~\ref{fig:model_comparison}. The results indicate that the model \texttt{some-model} consistently outperforms the other evaluated models. Cosine similarity was used for all similarity-based retrieval experiments.
