\begin{abstract}
In this assignment we developed a preferential web crawler for the news site 24ur.com\cite{24ur} that collects pages relevant to a user‑specified topical query (for example, the conflict between Russia and Ukraine). Relevance is estimated by embedding page metadata, such as titles, summaries, and keywords, into a vector space using a pre‑trained language model, then computing cosine similarity between the page embeddings and an embedding of the query. The crawler is comprised of an SQL database server, a RESTful API server that exposes database operations, and a client that uses browser automation to extract page content and hyperlinks and applies machine‑learning scoring to prioritize crawl targets.
\end{abstract}

This report describes the design and implementation of a preferential web crawler developed to extract pages 
relevant to a user‑specified topical query within a chosen domain. For this assignment, the chosen domain 
is the Slovenian news website 24ur.com.

The crawler comprises several components: a headless browser HTTP downloader and renderer, 
a data extractor for hyperlinks and images, a duplicate detector, a preferential URL frontier 
and a PostgreSQL database accessed via a RESTful API. The system is designed to run multiple parallel workers.

Pages are prioritised according to their relevance to a user‑specified topical query (e.g. 
the conflict between Russia and Ukraine) by embedding page metadata into a natural‑language string 
that is then converted into a vector representation using a pre‑trained language model and ranked 
using cosine similarity. The implementation also enforces canonical URLs, respects robots.txt directives 
and crawl delay constraints, and records pages, images and binary files found within the domain in 
the database.

The following sections document implementation details, parameter choices, performance evaluation, 
and visualisations of the resulting data.

\section{Methodology}
\subsection{Crawler Architecture}
The crawler is implemented as a modular system consisting of a REST API, a database, and a client service. 
The entire application is deployed using Docker Compose, with service configuration and crawling parameters defined in the docker compose configuration file.

\subsection{API Server and Database}
The REST API service is implemented using a Python Flask server. It utilizes the SQLAlchemy library to define and initialize the database 
schema in accordance with the schema provided in the assignment instructions.

The server supports configurable HTTP Basic Authentication and uses Flask’s built-in ad hoc SSL context to 
enable HTTPS, ensuring secure communication between components.

The crawler client interacts with the API using an api wrapper python module, which sends requests 
using the requests library.

\subsection{Crawler client}
The crawler client uses the threading module to spawn a specified number of worker threads, 
which process URLs from a shared URL frontier. 

Once a URL is acquired, it is first validated. Then, an instance of a Selenium \cite{selenium} Firefox headless web browser is launched. 
The target page is fetched and processed, and the relevant metadata is stored in the database.

Links found on the page are extracted, scored, and added to the frontier. This process repeats until the desired number of pages has been crawled. 
If an insertion into the database fails due to URL or content hash uniqueness constraints, the client handles the error by retrieving the pre-existing

The crawler respects the \texttt{robots.txt} file of each domain by ignoring links disallowed by the \texttt{Disallow} directive and adhering 
to the specified crawl delay.

\subsection{Page parsing}
Each subpage is parsed using the \texttt{BeautifulSoup} module. The parser is used to extract article keywords and
link URLs to other pages (in the \texttt{href} property or \texttt{onclick} JavaScript event handler).

The \texttt{onclick} event handler is processed by extracting URL candidates with regular-expression patterns 
that detect common JavaScript redirects as well 
as embedded JSON-like fields (\texttt{Url:}, \texttt{PlaybackUrl:}); if none match, it falls back to any quoted 
\texttt{.html} substring and returns the list of candidate URLs.

When saving a page to the database, each extracted URL is canonicalized using \texttt{urllib.parse} and \texttt{tldextract}. 
Redirection links belonging to the domain \texttt{go-ctr-tracker.pub.24ur.si}
are resolved using the \texttt{redir} key in the query string, the host is reduced to the 
registrable domain (\texttt{domain.suffix}), the path is normalized 
(collapsing \texttt{.}/\texttt{..} segments and removing trailing slashes), common tracking parameters 
(UTM, \texttt{fbclid}, \texttt{gclid}, \texttt{yclid}) are removed, remaining query parameters are sorted 
deterministically, and the final URL is returned with the \texttt{https} scheme and without fragments.

Binary resources are detected by file extension (\texttt{.pdf}, \texttt{.doc}, \texttt{.docx}, \texttt{.ppt}, \texttt{.pptx}). 
For HTML pages, all outgoing \texttt{<a href>} targets are canonicalized and then checked for these extensions; matching targets 
are not added to the link frontier, but are instead stored as \texttt{page\_data} entries with \texttt{data\_type\_code} set to 
the corresponding type (\texttt{PDF}/\texttt{DOC}/\texttt{DOCX}/\texttt{PPT}/\texttt{PPTX}), de-duplicated per page.

\subsection{Preferential Crawling}
- from page extract metadata
- using metadata construct a string with the format "Naslovi: {titles}. Povzetki: {summaries}. Ključne besede: {keywords}"
- use the generated natural language string to embed metadata into a BERT encoding vector (uses model "paraphrase-multilingual-MiniLM-L12-v2" \cite{BERTmodel})
- at the start of crawling embed the query sentence with the same model
- use cosine similarity as a measure of how well a link matches our query, links with no metadata get a priority of 0 (worst priority)
